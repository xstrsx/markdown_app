# DOCX Export Design

## Scope

Add Markdown to DOCX export for Android and Windows. The only UI entry is the
history page file menu. The editor page and its existing preview, including
`flutter_math_fork`, remain unchanged. No HTML export feature or external
executable is introduced.

## Architecture

`lib/export/markdown_ast_parser.dart` parses Markdown with the official
`markdown` package and produces an export-specific node model. Standard block
nodes include headings, paragraphs, lists, emphasis, blockquotes, code,
tables, and images. Inline and block LaTeX, plus Mermaid fenced blocks, are
represented explicitly and collected into one render queue.

`lib/export/headless_webview_render.dart` exposes a renderer abstraction with
`renderToSvg(type, content)`. Its default implementation loads offline assets
from `assets/render_assets`, communicates through the InAppWebView JS bridge,
and catches all asynchronous failures. Android creates a
`HeadlessInAppWebView` only for an export render and disposes it immediately
after the result. Windows checks WebView2 before rendering and reports a
dedicated unavailable-runtime state.

`lib/export/export_docx.dart` converts the export AST into a DOCX document.
Normal nodes become native Word paragraphs, headings, lists, tables, and code
blocks. Rendered SVGs become embedded images through `convert_svg`; inline
images are sized for a line, block formulas are centered, and Mermaid images
are constrained to the page width. Failed LaTeX becomes source text and failed
Mermaid becomes `图表渲染失败`, both with export warnings.

The generated bytes are saved through the existing `FileService.saveBytesAs`
flow, using the DOCX MIME type and a `.docx` default name.

## Dependencies and Assets

Add `docx`, `flutter_inappwebview`, and `convert_svg`, and update `markdown` to
the requested compatible version. Add only local offline render assets:
`render.html`, `katex.min.js`, `katex.min.css`, and `mermaid.min.js` under
`assets/render_assets`. No CDN or JavaScript embedded in Dart strings is used.

## Failure Handling

Rendering failures never abort an entire export. Windows without WebView2
shows the specified runtime warning at the export UI boundary and produces a
text-only DOCX for affected features. Save cancellation is not reported as a
failure. Unexpected errors are logged without exposing stack traces directly
to users.

## Verification

Tests cover AST classification and render queue behavior, deterministic DOCX
generation with an injected renderer, LaTeX and Mermaid fallback behavior,
WebView2-unavailable handling, and DOCX save metadata. Verification runs
targeted Flutter tests and `flutter analyze`; local Android/Windows packaging
is intentionally excluded.
